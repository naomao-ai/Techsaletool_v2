# 다중 사용자 환경에서의 보안(편집 충돌 방지) 및 로그 관리 로직 분석

이 데스크톱 애플리케이션은 클라우드 서버나 RDBMS 없이 공유 폴더 하나만을 매개로 다수의 사용자가 동시에 접근하여 데이터를 수정합니다. 이로 인해 발생할 수 있는 데이터 덮어쓰기(병행성 문제) 및 보안/무결성 이슈를 처리하기 위해 다음과 같은 로직이 구축되어 있습니다.

## 1. 다중 사용자 편집 잠금 (Locking) 보안 로직

웹 서버를 이용한 동기화와 달리, 이 앱은 **로컬 파일 시스템(디스크 I/O)을 이용한 분산 락(Distributed File Lock) 메커니즘**을 사용합니다. 

### 작동 원리 (Rust 백엔드 `src-tauri/src/main.rs`)
1. **락 폴더(`locks/`) 구성**: 데이터 파일 위치 상위에 `locks` 디렉터리를 만들고 특정 항목(아이템) 편집 시 해당 항목의 ID를 딴 `.lock` 파일을 무조건 생성합니다. (예: `item_tk123.lock`)
2. **잠금 획득 (`acquire_item_lock`)**:
   - 사용자가 세부 아이템 팝업을 열거나 항목을 더블클릭하여 편집 모드에 진입하려 할 때 트리거됩니다.
   - 이미 해당 항목에 대한 `.lock` 파일이 존재한다면, 내부에 적힌 사용자 ID와 본인의 ID를 대조합니다.
   - 타인이 점유 중이고 TTL(유지 시간)이 만료되지 않았다면 백엔드 단에서 에러(`Locked by [사용자명]`)를 반환하여 편집창 진입을 원천 차단합니다.
   - 선점자가 없다면 본인의 사번과 이름, 현재시간(`acquiredAt`), 만료시간(`ttlSeconds`: 600초)을 JSON으로 락 파일에 기록하여 소유권을 확보합니다.
3. **잠금 해제 및 좀비 락 방지 (`get_active_locks`, `release_item_lock`)**:
   - 편집을 마치고 창을 닫으면 명시적으로 `release_item_lock` 훅을 호출하여 자신의 락 파일을 삭제합니다.
   - 만약 앱 비정상 종료 등으로 락이 지워지지 않은 경우, 백엔드의 `get_active_locks`가 폴더 내 모든 락 파일들의 `ttlSeconds`를 순회하며 유통기한이 지난 락 파일을 스스로 가비지 컬렉팅(폐기 자동 삭제)하여 영구 잠금이 발생하는 것을 막습니다.

## 2. 실시간 감시 파이프라인 및 충돌 병합(Conflict Resolution)

강제 락을 걸지 않은 일반 작업(드래그 앤 드롭 이동 등) 중 우연히 동시에 두 사용자가 마스터 JSON 파일에 Write를 수행할 때 발생하는 데이터 유실 보안 대비입니다.

### 작동 원리 (`notify` 크레이트 & `autoMerge`)
1. **파일 와쳐 (File Watcher)**: Rust의 OS 파일시스템 감시 도구인 `notify` 크레이트가 작동하여 마스터 데이터 파일의 수정 시간을 모니터링합니다. 타인이 파일을 저장하면 100ms 디바운스 이내에 프론트엔드로 `shared-file-changed` 이벤트를 즉각 브로드캐스트합니다.
2. **충돌 식별**: 내 화면에서 내가 편집 중인데, 백그라운드 이벤트로 타인의 변경 데이터가 수신되면 프론트엔드 단의 `lastModifiedTime` 값을 대조하여 누군가 먼저 덮어썼음을 감지합니다.
3. **3-Way Merge 병합 로직 (`src/syncUtils.ts`)**: 
   - 앱 단에서 사용자가 입력하던 내용(Current)과 타인이 덮어쓴 데이터(Remote)를 베이스 기준 최신 상태로 논리적 병합(autoMerge)을 시도합니다.
   - 만약 항목이 겹쳐서 기계적 병합이 불가능한 경우 화면에 `ConflictModal.tsx`를 띄워 사용자에게 수동으로 데이터를 취사선택하게 만들어 무결성을 보장합니다.
4. **저장 통제**: 충돌 난 상태에서는 병합을 완전히 끝내기 전까지 파일 덮어쓰기 기능이 마비되도록 차단합니다.

## 3. 포렌식 및 활동 로그 (Changelog) 기능 처리 프로세스

다수 사용자가 하나의 마스터 파일을 공유하기 때문에 '누가 변경하여 장애를 냈는지' 역추적할 수단이 필수적으로 필요합니다. 이를 위해 자체적인 파일 기반 Changelog 관리자를 운영합니다.

### 작동 원리 (`TauriWrapper.appendChangelog`)
1. **일 단위 JSONL (개행형) 분산 기록**:
   - 데이터를 저장하거나(Save) 충돌을 해소(Conflict_Resolve)할 때 변경 내역을 `changelogs/YYYY-MM-DD.jsonl` 형태로 저장합니다. 
   - JSON 행렬 전체를 한 번에 덮어쓰는 것이 아닌, Rust의 `std::io::Write`를 사용해 파일 끝에 안전하게 이어서 붙이는(Append) 방식으로 I/O 병목 및 데이터 깨짐(Corruption) 문제를 방지합니다.
2. **트레이스 정보 (프론트 단에서 주입)**: 
   - 입력 시 현재 로그인 된 사용자 이름(`userName`), 사번(`userId`) 그리고 텍스트 기반의 실질적 변경 행위 요약(`changes` 배열, WBS 항목 수 등)이 투입됩니다.
3. **가시화 (ChangelogViewer)**:
   - 각 클라이언트 앱은 우측 패널의 [변경 내역] 버튼을 통해 당일 날짜의 `.jsonl` 파일을 라인 단위로 읽어 JSON 오브젝트로 복원(Parse)한 뒤 화면에 타임라인 형태로 시각화합니다. 
   - 이 로그 시스템을 통해 악의적인 삭제나 휴먼 에러 발생 지점의 책임을 입증할 수 있습니다.


## 4. 동시 편집(Auto-Merge) 및 충돌 해결(Conflict Resolution) 상세 로직

데스크톱 앱(Tauri) 환경에서 다수의 사용자가 동일한 마스터 파일(공유 JSON)을 작업할 때 발생하는 동시성 문제를 해결하기 위해, 깃(Git)과 유사한 **3방향 병합(3-Way Merge)** 알고리즘이 구현되어 있습니다.

### 4.1. 서로 다른 항목(셀) 편집 시: 자동 병합 (Auto-Merge)
사용자 A와 사용자 B가 각각 다른 항목(셀)의 데이터를 변경하고 저장할 경우, 시스템은 충돌을 발생시키지 않고 두 변경사항을 논리적으로 결합합니다.

1. **버전 충돌 감지**: 사용자 A가 저장할 때, 백엔드는 파일의 최신 수정시간(`lastModifiedTime`)을 대조하여 누군가(사용자 B) 먼저 덮어썼음을 감지하고 `VERSION_CONFLICT` 예외를 발생시킵니다.
2. **3-Way Merge 수행 (`autoMerge`)**:
   - 프론트엔드는 이 예외를 캐치한 뒤 `syncUtils.ts`의 `autoMerge` 함수를 호출합니다.
   - `baseState`(편집을 시작할 때의 원본), `clientState`(내가 수정한 내용), `serverState`(저장소에 타인에 의해 새로 덮어씌워진 최신 내용) 3가지를 비교합니다.
   - 내가 수정한 필드와 상대방이 수정한 필드가 겹치지 않는다면, 시스템은 두 변경점을 하나로 모은 `mergedState`를 생성합니다.
3. **병합 성공 및 알림**: 병합이 안전하게 끝나면, 별도 경고창 없이 데이터가 갱신되고 화면에 조용히 성공 알림(Toast)만 노출합니다.

**관련 코드 (`src/App.tsx` 중 자동 병합 처리 부분):**
```typescript
try {
  // 타인에 의해 파일 변경이 발생했는지 시간값 비교 후 저장 요청
  await TauriWrapper.saveToSharedFolder(tauriFilePath, dataToSave, baseStateRef.current.lastModifiedTime);
} catch (err: any) {
  if (err === 'VERSION_CONFLICT') {
    // 1. 최신 서버(공유폴더) 데이터 다시 읽어오기
    const result = await TauriWrapper.readFromSharedFolder(tauriFilePath);
    const serverState = normalizeState(JSON.parse(result.data));
    serverState.lastModifiedTime = result.lastModified || Date.now();
    
    // 2. 3-Way 병합 시도 (내가 바꾼것과 타인이 바꾼 것 조율)
    const mergeResult = autoMerge(data, serverState, baseStateRef.current);
    
    if (mergeResult.success && mergeResult.mergedState) {
      // 3. 충돌 없는 안전한 병합 성공 시 즉시 갱신
      setState(normMerged);
      baseStateRef.current = normMerged;
      setToast({ message: '다른 사용자의 변경사항과 안전하게 병합되었습니다.', type: 'success' });
    } else {
      // 4. 같은 항목/셀을 수정하여 병합 불가능한(충돌) 경우 모달 표시
      setConflictData({
        serverState: serverState,
        clientState: data,
        baseState: baseStateRef.current
      });
      setIsConflictModalOpen(true);
    }
  }
}
`


### 4.2. 같은 항목(셀) 동시 편집 시: 충돌 해결 경고창 (Conflict Modal)
사용자 A와 사용자 B가 **완전히 동일한 항목의 같은 필드(예: 동일한 WBS의 작업내용 셀)**를 동시에 수정하여 저장하려고 하면, 시스템은 기계적인 덮어쓰기를 포기하고 사용자에게 선택권을 넘깁니다.

1. **병합 실패 (Conflict)**: `autoMerge` 함수가 같은 필드에 상충되는 변경사항을 감지하면 `success: false`를 반환합니다.
2. **ConflictModal 렌더링**: 화면에 팝업창(충돌 해결창)이 나타나며, 외부에서 변경된 데이터(타인의 내용)와 현재 편집 중인 데이터(나의 내용)를 나란히 비교하여 보여줍니다.
3. **사용자의 선택 수용**:
   - `[내 데이터로 덮어쓰기]`: 상대방의 변경사항을 무시하고 나의 작업물을 강제로 덮어씌웁니다.
   - `[서버 데이터 유지]`: 내 변경사항을 폐기하고 상대방(공유폴더)의 최신 데이터를 화면에 반영합니다.

**관련 로직 (`src/ConflictModal.tsx` 기반 동작 흐름):**
```typescript
// ConflictModal에서 사용자가 선택을 마치면 호출되는 콜백 로직
const handleResolve = (resolvedState: AppState) => {
  setIsConflictModalOpen(false);
  
  // 사용자가 취사선택한 최종 State를 화면에 적용
  setState(resolvedState);
  baseStateRef.current = resolvedState;
  
  // 해결된 상태를 다시 덮어쓰기 로직으로 넘겨 공유 파일에 Force Save 진행
  forceSaveAfterConflict(resolvedState);
};
```
