# 작업 지시서 — 셀 클릭 잔여 지연 해소 (React.memo 스위치 ON + 행별 prop 정리)

| 항목 | 내용 |
|------|------|
| **대상 파일** | `src/components/Spreadsheet.tsx` (v2.1, 약 2,671줄) |
| **증상** | 최적화로 지연 대폭 감소했으나, 500행 이상에서 클릭 시 미세 지연 잔존 (사용자 거부감) |
| **환경** | 단일 HTML → 오프라인 이동 → Tauri 데스크톱앱 빌드 |
| **제약** | **신규 npm 패키지 추가 금지** (오프라인 빌드). 아래 작업은 모두 React 내장 기능 + CSS만 사용 |
| **목표** | 클릭/드롭다운/잠금 갱신 시 리렌더 행을 500+개 → 1~2개로 축소, 대용량 DOM 묵직함 제거 |

## 0. 핵심 진단 (작업 배경)

기존 최적화(`useCallback` 13개, `reqById` Map, `filteredAndSorted` useMemo, `isActive`/`activeField` 전달)는 **각 행의 렌더 비용을 낮추는 데까지만** 성공했다. 그래서 체감이 좋아졌다.

그러나 **여전히 500행이 매 렌더마다 전부 다시 그려진다.** 원인은 명확하다.

> **`SpreadsheetRow`(line 1990)가 순수 함수이며 `React.memo`로 감싸여 있지 않다.**
> memo 게이트가 없으므로 React는 props 비교조차 하지 않고, 부모가 리렌더될 때마다 자식 500개를 무조건 다시 렌더한다.

지금까지의 모든 작업이 "memo를 켜기 위한 준비"였는데, **정작 스위치(`React.memo`)를 켜지 않은 상태**다.

**작업 우선순위**

| 순위 | 작업 | 효과 | 필수 여부 |
|:---:|------|------|:---:|
| 1 | `React.memo` 래퍼 추가 | 클릭 지연 대부분 해소 | **필수** |
| 2 | 공유 객체/ID prop → 행별 원시값 | 드롭다운·잠금 갱신 시 전 행 리렌더 차단 | **필수** |
| 3 | formula 함수 컴파일 캐싱 | 초기/편집 렌더 비용 감소 | 권장 |
| 4 | `content-visibility: auto` | 대용량 DOM 스크롤·초기 렌더 묵직함 제거 | 권장 |

## Task 1. `React.memo` 래퍼 추가 ★필수 (잔여 지연의 대부분)

준비가 모두 끝나 있으므로 래핑만 하면 된다. 클릭 시 리렌더가 500행 → 2행으로 즉시 축소된다.

### 위치: line 1990

```js
// 변경 전
function SpreadsheetRow({
  req, isSelected, dragOverRowId, columns, /* … 기존 props … */
}: any) {
  /* 본문 */
}

// 변경 후
const SpreadsheetRow = React.memo(function SpreadsheetRow({
  req, isSelected, dragOverRowId, columns, /* … 기존 props … */
}: any) {
  /* 본문 그대로 */
});
```

> 본문은 단 한 줄도 수정하지 않는다. 함수 선언을 `const … = React.memo(function … { … });` 형태로 감싸기만 한다.

## Task 2. 공유 객체/ID prop을 행별 원시값으로 변경 ★필수

memo를 켜도, **모든 행에 동일한 값으로 전달되는 prop**이 바뀌면 그 순간 500행이 또 전부 리렌더된다. `isActive`에서 이미 적용한 패턴(행별 boolean)을 아래 prop들에도 동일하게 확장한다.

### 2-1. 호출부 수정 (line 1491 부근)

```jsx
// 변경 전
<SpreadsheetRow
  ...
  activeLocks={activeLocks}
  showPriorityDropdownId={showPriorityDropdownId}
  showAssigneeDropdownId={showAssigneeDropdownId}
  showStatusDropdownId={showStatusDropdownId}
  ...
/>

// 변경 후
<SpreadsheetRow
  ...
  isLockedByOther={!!activeLocks[req.id] && activeLocks[req.id].userId !== currentUser?.id}
  lockedByName={activeLocks[req.id]?.userName ?? null}
  isPriorityOpen={showPriorityDropdownId === req.id}
  isAssigneeOpen={showAssigneeDropdownId === req.id}
  isStatusOpen={showStatusDropdownId === req.id}
  ...
/>
```

### 2-2. `SpreadsheetRow` 내부 수정

- props 구조분해에서 `activeLocks`, `showPriorityDropdownId`, `showAssigneeDropdownId`, `showStatusDropdownId` 제거
- 새 props `isLockedByOther`, `lockedByName`, `isPriorityOpen`, `isAssigneeOpen`, `isStatusOpen` 추가
- 내부 치환 (예시):

| 변경 전 | 변경 후 |
|---------|---------|
| `activeLocks[req.id] && activeLocks[req.id].userId !== currentUser?.id` (line 2100, 2150) | `isLockedByOther` |
| `activeLocks[req.id].userName` (line 2108, 2114, 2117, 2167) | `lockedByName` |
| `showPriorityDropdownId === req.id` | `isPriorityOpen` |
| `showAssigneeDropdownId === req.id` | `isAssigneeOpen` |
| `showStatusDropdownId === req.id` | `isStatusOpen` |

> 효과: 잠금 폴링이 갱신되거나 드롭다운 하나가 열려도, 값이 실제로 바뀌는 행 1개만 리렌더된다. 나머지 행은 원시값 비교가 통과되어 건너뛴다.

## Task 3. formula 함수 컴파일 캐싱 (line 2468)

```js
const func = new Function('req', 'today', `return ${col.formula || '""'}`);
```

수식 셀이 렌더마다 자바스크립트 함수를 문자열에서 새로 컴파일한다. memo 적용 후엔 리렌더되는 행만 부담하지만, **초기 렌더(500행 전부)와 편집 시** 비용이 남는다. 수식 문자열을 키로 캐싱한다.

```js
// 컴포넌트(또는 모듈) 상단
const formulaCache = useRef(new Map());
const getFormulaFn = (f) => {
  if (!formulaCache.current.has(f)) {
    formulaCache.current.set(f, new Function('req', 'today', `return ${f || '""'}`));
  }
  return formulaCache.current.get(f);
};

// line 2468 교체
const func = getFormulaFn(col.formula);
```

> 보안 참고: `new Function`은 임의 코드 실행 위험이 있다. 폐쇄망 내부 도구라 위험도는 낮으나, 중장기적으로 수식 파서 대체를 권장한다.

## Task 4. 대용량 DOM 묵직함 제거 — `content-visibility` (무의존)

Task 1~3으로 "클릭 지연"은 사라진다. 다만 행이 많을 때 남는 **스크롤·초기 렌더의 묵직함**은 DOM에 7,500개(500행 × 15열) 셀이 항상 존재하기 때문이다. 패키지 없이 CSS로 해결한다.

```css
/* 각 행(tr) 또는 행 컨테이너에 적용 */
.spreadsheet-row {
  content-visibility: auto;
  contain-intrinsic-size: 40px;   /* 실제 행 높이에 맞게 조정 */
}
```

- Tauri의 WebView2(Chromium)가 화면 밖 행의 레이아웃·페인트를 자동으로 건너뛴다.
- **신규 의존성 전혀 없음.** 단일 HTML / Tauri 빌드 그대로 통과.
- 가상화 라이브러리(`@tanstack/react-virtual`)는 **수천 행 이상**으로 커질 때만 고려 (그 경우 오프라인 패키지 전송 필요).

## 주의사항

- **Task 1이 빠지면 Task 2~4는 효과가 제한적이다.** memo 게이트가 없으면 props를 아무리 정리해도 React가 비교 자체를 하지 않는다.
- Task 2에서 **공유 ID/객체를 그대로 넘기면 안 된다.** 반드시 `=== req.id` 비교로 행별 boolean을 만들어 전달한다. (이미 `isActive`에서 검증된 패턴)
- `content-visibility`는 행 높이가 들쭉날쭉하면 `contain-intrinsic-size`를 평균 높이로 맞춰야 스크롤 점프가 줄어든다.
- 본 작업은 전부 React 내장 API(`React.memo`/`useRef`)와 표준 CSS만 사용하므로 오프라인 빌드 환경에 영향이 없다.
