## 0. 핵심 진단 (작업 배경)

`SpreadsheetRow`를 `React.memo`로 분리(line 1700)했으나 **memo가 한 번도 작동하지 않는 상태**다. 두 가지가 원인이다.

1. **행에 전달하는 핸들러 10여 개가 `useCallback`이 아님** → 매 렌더마다 새 함수 참조 생성 → 모든 행의 props가 "변경됨"으로 판정.
2. **`activeCellEditor` 객체 전체를 모든 행에 전달**(line 1324) → 클릭마다 새 객체 → 모든 행 props 변경.

→ `React.memo`의 기본 얕은 비교가 매번 실패 → 클릭 한 번에 500행 전부 리렌더 → 1초 지연.

**작업 우선순위: Task 1 → Task 2 가 필수(이 둘로 증상 해소). Task 3·4는 보조/방어.**

### 1-1. 일반 핸들러 (의존성 단순)

대부분 `setRequirements(prev => …)` 함수형 업데이트라 외부 의존성이 없다. 로직은 그대로 두고 `useCallback`으로만 감싼다.

```js
// 변경 전
const updateRequirementField = (rowId, field, value) => { /* 기존 로직 */ };

// 변경 후
const updateRequirementField = useCallback((rowId, field, value) => {
  setRequirements(prev => prev.map(req => { /* 기존 로직 그대로 */ }));
}, [setRequirements]);
```

동일 패턴 적용 대상 (deps는 대부분 `[]` 또는 `[setRequirements]`):

```js
const handleSelectRow     = useCallback((id, checked) => { /* 기존 로직 */ }, [setSelectedIds]);
const handleDuplicateRow  = useCallback((sourceRowId, insertAfterId) => { /* 기존 로직 */ }, [setRequirements]);
const handleRowDragStart  = useCallback((e, rowId) => { /* 기존 로직 */ }, []);
const handleRowDragOver   = useCallback((e, rowId) => { /* 기존 로직 */ }, []);
const handleRowDrop       = useCallback((e, targetRowId) => { /* 기존 로직 */ }, [setRequirements]);
const handleGridPaste     = useCallback((e, startRowId, startColId) => { /* 기존 로직 */ }, [/* 필요한 setter */]);
```

> 각 핸들러 내부에서 참조하는 **setter 외의 외부 변수**가 있으면 그것만 deps에 추가한다. setter 함수(`setX`)는 React가 안정성을 보장하므로 deps에 넣어도 무방하다.

### 1-2. `setActiveCellEditor` — ref 패턴 필수 (가장 중요)

이 함수는 내부에서 `activeCellEditor`, `activeLocks`를 읽는다. 이를 deps에 넣으면 **클릭마다 함수가 새로 생성되어 memo가 다시 깨진다.** 최신값은 ref로 읽어 함수 정체성을 고정한다.

```js
// 컴포넌트 상단에 ref 추가
const cellRef  = useRef(activeCellEditor);
const locksRef = useRef(activeLocks);
cellRef.current  = activeCellEditor;   // 매 렌더 최신값 동기화
locksRef.current = activeLocks;

// setActiveCellEditor 를 useCallback + ref 로 재작성
const setActiveCellEditor = useCallback(async (val) => {
  const current = cellRef.current;        // ← activeCellEditor 대신 ref 사용
  const locks   = locksRef.current;        // ← activeLocks 대신 ref 사용

  // 닫기: 기존 release_item_lock 로직 (current 사용)
  // 열기: 낙관적 setActiveCellEditorState(val) 먼저 → acquire_item_lock
  //       실패 시 locks 로 잠금자 이름 표시
  // ... 기존 본문 로직을 current / locks 로 치환 ...
}, [dbPath, currentUser]);   // 자주 바뀌지 않는 값만 deps
```

**검증 포인트:** 이 함수가 클릭마다 새로 만들어지지 않아야 한다. 이것이 memo 실패의 주범이었다.

## Task 2. `activeCellEditor` 객체 → 행별 boolean 전달 ★필수

### 2-1. 호출부 수정 (line 1324 부근)

```jsx
// 변경 전
<SpreadsheetRow
  ...
  activeCellEditor={activeCellEditor}
  ...
/>

// 변경 후
<SpreadsheetRow
  ...
  isActive={activeCellEditor?.rowId === req.id}
  activeField={activeCellEditor?.rowId === req.id ? activeCellEditor.field : null}
  ...
/>
```

### 2-2. `SpreadsheetRow` 내부 수정 (line 1700 부근)

- props 구조분해에서 `activeCellEditor` 제거 → `isActive`, `activeField` 추가
- 내부의 모든 `activeCellEditor?.rowId === req.id` 비교를 `isActive` 로 치환
- 내부의 `activeCellEditor?.field === '...'` 비교를 `activeField === '...'` 로 치환

> 효과: 클릭 시 `isActive`가 `true`/`false`로 바뀌는 행은 직전 행·새 행 단 2개. 나머지 498개 행은 `false === false`로 비교를 통과해 리렌더를 건너뛴다.

## Task 3. rollup 조회 O(n²) → O(n) (보조, 편집 속도 개선)

`requirements={requirements}`를 모든 행에 넘겨 셀 내부에서 `requirements.find(...)`(line 1797·1814)로 조회하고 있다. 편집 시 전 행 리렌더의 원인이 된다.

```js
// 부모 컴포넌트
const reqById = useMemo(
  () => new Map(requirements.map(r => [r.id, r])),
  [requirements]
);
// SpreadsheetRow 에 reqById 전달, 내부 조회를 교체:
// requirements.find(r => r.id === rid)  →  reqById.get(rid)
```

## Task 4. (선택) 방어용 커스텀 comparator

Task 1·2를 완료하면 기본 비교로 충분하다. 안전망이 필요하면 명시적 비교를 추가한다. **단, Task 1(useCallback)이 선행되어야 stale closure 없이 안전하다.**

```js
const SpreadsheetRow = React.memo(Component, (p, n) =>
  p.req === n.req &&
  p.isActive === n.isActive &&
  p.activeField === n.activeField &&
  p.isSelected === n.isSelected &&
  p.dragOverRowId === n.dragOverRowId &&
  p.columns === n.columns &&
  p.columnWidths === n.columnWidths &&
  p.minimizedColumns === n.minimizedColumns &&
  p.activeLocks === n.activeLocks &&
  p.assigneesPool === n.assigneesPool
);
```

## 주의사항

- **순서 중요:** Task 1을 건너뛰고 Task 2만 적용하면 효과가 없다. 핸들러 참조가 매 렌더 바뀌어 memo가 계속 깨지기 때문이다.
- `useCallback` deps에 `activeCellEditor`·`activeLocks` 같은 **자주 바뀌는 값을 직접 넣지 말 것** → 반드시 ref로 우회.
- 핸들러 내부에서 `requirements`를 직접 참조하지 말고 `setRequirements(prev => …)` 함수형 업데이트를 유지할 것 → deps를 비워 함수 정체성을 안정화할 수 있다.
- 본 작업은 전부 React 내장 API(`useCallback`/`useRef`/`useMemo`/`React.memo`)만 사용하므로 오프라인 빌드 환경에 영향이 없다.
