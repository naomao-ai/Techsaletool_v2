# 작업지시서 (개정) — 요구조건 관리 플랫폼 v2.1.3 성능 개선

> **개정 사유**: 기존 작업지시서는 **WS 릴레이 + 호스트 선출(hostManager)** 구조를 전제로 작성되었으나,
> 실제 업로드 빌드(`...서버x`)는 **공유파일 + 파일와처 + 낙관적 버전락** 구조다.
> 본 문서는 실제 코드(라인 기준)에 맞춰 적용 가능한 작업만 남기고, 불가/완료 항목은 사유와 함께 명시한다.
>
> **대상 구조(실측)**: 공유 데이터파일(`save_data`로 원자적 쓰기) → `start_file_watcher`가 `shared-file-changed` emit → 타 클라이언트가 `read_data` + `executeSmartMerge`로 동기화. 충돌은 `expectedVersion` 낙관적 락으로 처리.
> **문제**: 데이터가 많아지면 셀 1개 편집에도 앱 전체가 버벅임.
> **근본 원인(실측)**: 편집 1회당 **전체 데이터셋을 여러 번 직렬화**(JSON.stringify N) + **전 행 리렌더**. 모든 비용이 변경량이 아니라 전체 크기 N에 비례.

## 0. 핵심 메시지 (작업 전 필독)

- **Phase 1~4는 전부 프론트엔드 변경**이며 **신규 npm/Cargo 의존성 0개**다. 내부망 빌드/전송 제약과 충돌하지 않는다.
  - 인터넷 환경에서 `vite build` → 산출 `index.html`만 내부망 전송 → 기존 크레이트 캐시로 Tauri 재컴파일(오프라인 가능).
  - `main.rs`, `Cargo.toml`, vendoring **불필요**(Phase 5 제외).
- **Phase 5(SQLite)만** 신규 Cargo 크레이트(`rusqlite`)가 필요 → vendoring 대상(배경문서 부록 B-2). 데이터가 계속 커질 때의 근본책이며, Phase 1~4로 먼저 버틴 뒤 측정으로 판단한다.
- 작업은 **위에서 아래 순서**로 진행한다. **Phase 1(직렬화 제거) + Phase 2(memo 복원)** 만 적용해도 체감 버벅거림의 대부분이 사라진다.

## 3. 상세 작업

### Phase 1 — 직렬화 폭발 제거 *(최우선·즉효, 기존 지시서 누락분)*

데이터가 커질 때 가장 무거운 지점은 memo보다 여기다. 편집 1회에 전체 데이터셋이 **3회** 직렬화된다.

#### 1-A. `dataPayload`를 매 렌더 → 데이터 변경 시로 (App.tsx:106)

**현상**: `App.tsx:106`에서 전체 직렬화가 메모 없이 **매 렌더마다** 실행된다. 락 폴링(`get_active_locks`), 토스트, 필터 입력 등 데이터와 무관한 리렌더에도 전체 N을 직렬화한다.

```ts
// before (App.tsx:106) — 매 렌더 실행
const dataPayload = JSON.stringify({ tabDataMap, tabs, assigneesPool, appName });

// after — 실제 데이터가 바뀔 때만 직렬화
const dataPayload = useMemo(
  () => JSON.stringify({ tabDataMap, tabs, assigneesPool, appName }),
  [tabDataMap, tabs, assigneesPool, appName]
);
```

> 저장 effect(`App.tsx:447`)의 의존성 배열에 이미 `dataPayload`가 있으므로 동작은 동일하게 유지되고, 비-데이터 리렌더에서의 불필요한 직렬화만 사라진다.

#### 1-B. undo 히스토리의 이중 직렬화 제거 (Spreadsheet.tsx:78)

**현상**: `setRequirements` 래퍼(`Spreadsheet.tsx:73-86`)가 매 편집마다 `JSON.stringify(prev) !== JSON.stringify(next)`로 변경 여부를 판단한다 → **편집 1회당 전체 배열 2회 추가 직렬화**.

```ts
// before (Spreadsheet.tsx:78)
if (JSON.stringify(prev) !== JSON.stringify(next)) {
  historyRef.current.push([...prev]);
  ...
}

// after — 참조 비교로 교체 (편집은 항상 새 배열을 만들므로 신뢰 가능)
if (prev !== next) {
  historyRef.current.push(prev);          // [...prev] → prev: 불변 갱신이라 얕은 보관으로 충분
  if (historyRef.current.length > MAX_HISTORY) historyRef.current.shift();
  redoStackRef.current = [];
}
```

> **트레이드오프**: 내용은 같지만 새 참조인 배열이 들어오면 히스토리에 1칸 더 쌓일 수 있다(최대 20칸 cap이라 무해, undo 시 시각적 no-op일 뿐). 정확성이 중요하면 길이/얕은 비교를 추가하되, **전체 `JSON.stringify` 비교는 절대 복원하지 않는다.**

**검증**: §2-③으로 편집 1회당 stringify 호출이 (106의 1회 또는 0회) 수준으로 줄었는지 확인.

### Phase 2 — `React.memo` 복원 (편집 1회 = 그 행만 리렌더)

**현상**: `SpreadsheetRow`(`Spreadsheet.tsx:2091`)는 `React.memo`지만 **비교자가 없고**, 모든 행에 넘기는 두 prop이 매 렌더 새 참조라 효과가 0이다.
- `getCellStickyStyle`(`:1175`) — 렌더 스코프 일반 함수 → 매번 새 함수.
- `reqById`(`:327`) — `requirements` 변경 시 `new Map(...)` → 매번 새 Map.

결과적으로 셀 1개를 바꿔도 **전 행이 리렌더**된다.

#### 2-A. `getCellStickyStyle`를 `useCallback`으로 안정화 (Spreadsheet.tsx:1175)

본문이 참조하는 외부값은 `frozenOffsets`뿐이다(`getStickyStyle`(`:1163`)은 인자만 쓰는 순수 함수, `frozenOffsets`(`:870`)는 이미 `useMemo`). 따라서 의존성은 `[frozenOffsets]`만으로 정확하다.

```ts
// before: const getCellStickyStyle = (colId, isHeader = false) => { ... }
const getCellStickyStyle = useCallback(
  (colId: string | 'index' | 'checkbox', isHeader = false): React.CSSProperties => {
    // ... 기존 본문 그대로 ...
  },
  [frozenOffsets]   // ← 기존 지시서의 [columnWidths, minimizedColumns, frozenColumnId]는 현행 코드와 불일치. frozenOffsets 하나로 충분(그 안에 이미 포함).
);
```

> 참고: `frozenOffsets`는 `[frozenColumnId, columns, minimizedColumns, columnWidths]`가 바뀔 때만 갱신되므로, 동결/너비/최소화 변경 시에만 새 함수가 되고 그때만 행이 갱신된다.

#### 2-B. `SpreadsheetRow`에 커스텀 비교자 추가 (Spreadsheet.tsx:2091)

행 자신의 표시에 영향을 주는 prop만 비교하고, 콜백/refs/`reqById`는 제외한다. 현행 호출부(`:1535-1574`)·파라미터(`:2091-2128`)에 실제로 존재하는 prop명에 맞췄다.

```ts
const SpreadsheetRow = React.memo(function SpreadsheetRow({ /* ...기존 구조분해... */ }: any) {
  // ... 기존 본문 그대로 ...
}, (prev, next) => {
  return (
    prev.req === next.req &&                       // 이 행 데이터(참조 비교)
    prev.isSelected === next.isSelected &&
    prev.isActive === next.isActive &&
    prev.activeField === next.activeField &&
    prev.isLockedByOther === next.isLockedByOther &&
    prev.lockedByName === next.lockedByName &&
    prev.isPriorityOpen === next.isPriorityOpen &&
    prev.isAssigneeOpen === next.isAssigneeOpen &&
    prev.isStatusOpen === next.isStatusOpen &&
    prev.dragOverRowId === next.dragOverRowId &&
    prev.columns === next.columns &&
    prev.minimizedColumns === next.minimizedColumns &&
    prev.columnWidths === next.columnWidths &&
    prev.assigneesPool === next.assigneesPool &&
    prev.currentUser === next.currentUser &&
    prev.getCellStickyStyle === next.getCellStickyStyle  // 2-A 덕에 안정화됨
    // reqById 의도적 제외(아래 트레이드오프), 콜백/refs 제외(식별자 변경이 행 표시를 바꾸지 않음)
  );
});
```

> **트레이드오프(중요)**: 비교자에서 `reqById`를 제외하므로, 관계(relation) 컬럼이 *다른* 행의 변경을 즉시 반영하지 못하고 해당 행 자체가 갱신될 때 따라온다(`Spreadsheet.tsx:2616,2632`의 `reqById.get(...)`). 관계 컬럼은 드물게 쓰이므로 일반적으로 수용 가능. 실시간성이 필요하면 관계 셀만 별도 구독/컨텍스트로 분리한다.

#### 2-C. 상위 콜백 안정화 (App.tsx:846) — 선택이지만 권장

**현상**: `App.tsx:846`의 `setRequirements={(reqs) => setTabDataMap(...)}`가 **인라인 화살표라 매 렌더 새 참조**다. 이 때문에 `Spreadsheet`의 `setRequirements` 래퍼(deps `[originalSetRequirements]`)도 매 렌더 재생성된다(비교자에서 콜백을 제외하므로 행 리렌더에는 직접 영향 없지만, `Spreadsheet` 내부 `useCallback`들의 안정성이 깨진다).

```ts
// App 컴포넌트 스코프에서 useCallback으로 승격 (currentTab 의존)
const handleSetRequirements = useCallback((reqs: React.SetStateAction<Requirement[]>) => {
  setTabDataMap(prev => ({
    ...prev,
    [currentTab]: {
      ...(prev[currentTab] || { columns: DEFAULT_COLUMNS }),
      requirements: typeof reqs === 'function' ? (reqs as any)(prev[currentTab]?.requirements || []) : reqs,
    },
  }));
}, [currentTab]);
// ...
<Spreadsheet requirements={tData.requirements} setRequirements={handleSetRequirements} ... />
```

> 함수형 업데이트 형태를 유지해 동시 편집 시 stale state를 방지한다. 기존 인라인이 값 형태만 받았다면 위처럼 함수형도 처리하도록 맞춘다.

**검증**: Profiler "Highlight updates"로 셀 1개 편집 시 **편집한 행만 하이라이트**되는지 확인. Phase 1과 합치면 대용량에서도 편집이 즉각 반영된다.

### Phase 4 — 저장 디바운스 상향 + 종료 플러시 *(보수적 적용)*

**위치**: `App.tsx:447-510`(저장 effect), `:509`의 `setTimeout(syncData, 500)`.
**주의(아키텍처 특성)**: 이 구조에서 `save_data`(전체 파일 쓰기)는 **타 클라이언트 동기화 채널**이다(상대 와처가 이 쓰기를 감지해 read+merge). 따라서 파일 쓰기를 과도하게 늦추면 동기화 지연이 커진다. **기존 지시서의 "스냅샷 주기 분리/델타 전환"은 와처·머지 경로까지 함께 바꿔야 하므로 본 Phase 범위에서 제외**하고, 안전한 두 가지만 적용한다.

```ts
// 1) 디바운스 상향: 연속 편집 중 전체 쓰기 빈도 감소 (입력 응답성과 무관, 저장은 백그라운드)
const timer = setTimeout(syncData, 1000); // 500 → 1000ms (동기화 지연 허용치 내에서 조정)

// 2) 종료/탭 전환 시 강제 1회 flush (마지막 변경 유실 방지)
useEffect(() => {
  const flush = () => {
    try { localStorage.setItem('offline_db', dataPayloadRef.current); } catch {}
    // dbPath가 있으면 save_data 동기 보장은 어려우므로 localStorage 우선 보존
  };
  window.addEventListener('beforeunload', flush);
  document.addEventListener('visibilitychange', () => { if (document.hidden) flush(); });
  return () => window.removeEventListener('beforeunload', flush);
}, []);
```

> `dataPayload`는 effect 안에서 1회만 만들어 localStorage·파일에 재사용 중인 구조를 유지한다(중복 stringify 추가 금지). flush용으로 최신 payload를 `useRef`에 담아둔다.

**검증**: 연속 타이핑 중 디스크 쓰기 횟수 감소, 종료/전환 시 마지막 상태 보존. 다중 사용자에서 동기화 지연이 허용 범위인지 §2-⑤로 확인 후 디바운스 값 미세 조정.

### Phase 5 — (근본책) 단일 거대 JSON → 임베디드 DB(SQLite) *선택*

**언제**: Phase 1~4 적용 후에도 데이터 증가로 한계가 측정될 때.
**효과**: 행 단위 read/write/정렬/인덱스를 DB가 O(log N)으로 처리. "전체 직렬화/전체 파일 쓰기" 모델 자체를 제거.

**내부망 제약(반드시 준수, 배경문서 부록 B-2)**:
- `rusqlite`(+`libsqlite3-sys`)는 **신규 Cargo 크레이트 + C 컴파일**. 내부망 캐시에 없으면 빌드 차단.
- 인터넷 환경에서 `cargo vendor`로 의존성 동봉 후 통째 전송:
  - `src-tauri/.cargo/config.toml`에 vendored-sources 설정.
  - 전송 대상: `vendor/`, `.cargo/config.toml`, 변경된 `Cargo.toml`/`Cargo.lock`, `main.rs`.
- 보고 시 "추가 크레이트(`rusqlite` 버전), 표준 라이브러리로 대체 불가 사유, vendor 크기 비용" 명시.

**마이그레이션 스케치**:
1. 스키마: `requirements(tab_id, id, title, priority, status, due_date, custom_json, sort_index, updated_at)`, `columns(tab_id, ...)`, `tabs(...)`.
2. 커맨드 교체: 현행 `read_data`/`save_data`(전체) → `query_requirements(tab_id, filter, sort, page)`, `upsert_requirement(...)`, `delete_requirement(...)` 등 행 단위. **단, 와처·smart merge·낙관적 락(`expectedVersion`) 로직을 행 단위 모델로 재설계해야 함**(현행 동시성 모델의 핵심이므로 가장 큰 작업).
3. 기존 단일 JSON을 1회 임포트하는 마이그레이션 커맨드 제공.
4. 프론트는 탭/페이지 단위 로드(전체 메모리 적재 제거) → `tabDataMap` 통짜 보관·직렬화 자체가 사라짐 → Phase 1의 직렬화 병목 원천 소멸.

> 비용·재설계 범위가 가장 크므로 **Phase 1~4로 부족함이 측정으로 확인된 뒤** 착수한다.
