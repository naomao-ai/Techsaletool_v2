**아키텍처 요지 (다중 사용자):** 이 앱의 다중 사용자 협업은 두 갈래다.
- **웹/서버 모드** — Express + **Socket.IO 락** (`server.ts`)
- **Tauri 네이티브(오프라인) 모드** — 중앙 서버 없이 **공유 드라이브 JSON 파일 + 파일 락 + 3-way 병합**(`main.rs`, `App.tsx executeSmartMerge` → `mergeEngine.ts`)

### 결함 #3 — 동시 저장 시 mtime 버전 게이트가 lost update를 못 막음 (위험 2)
- **발견:** 2026-07-03 (L2)
- **심각도:** 🔴 High (다중 사용자 동시 저장 시 한쪽 데이터 완전 유실)
- **상태:** ✅ **수정 완료 (2026-07-03, §7)** — §4 설계 개선안 #3(A+B 조합)대로 구현. 상세는 [§7.1](#71-결함3-수정--rev-기반-cas--임계구역-락)
- **재현 방법 (Rust 테스트, `l2_tests`):**
  - `risk2_concurrent_same_version_causes_lost_update`: A·B가 같은 버전 v0을 읽고 500ms 이내 순차 저장 → B가 stale v0인데도 충돌 없이 통과, A의 저장을 덮음. 최종 파일에 A 없음(유실).
  - `risk2_threaded_no_mutual_exclusion_between_instances`: 두 스레드(=두 인스턴스)가 같은 v0로 동시 저장 → 둘 다 `Ok`(상호배제 부재), 최종 파일은 한쪽만 남음(병합 아님).
  - 대조군 `control_version_gate_rejects_when_gap_exceeds_tolerance`: 간격이 500ms를 넘으면 정상적으로 `VERSION_CONFLICT` 발생(게이트 자체는 부분 작동).
  - **네이티브 L3 실추가 증거 (2026-07-03, §5.8):** 실제 2-인스턴스 `.exe`로 A·B가 같은 필드를 동시 편집 시켰을 때, 매 회 한쪽만 반영되는 lost update가 실제로 재현됨. 한 회차에서는 유실된 쪽 콘솔에 `Tauri save failed Failed to rename tmp to original after retries`(=`main.rs` L90-100의 rename 5회 재시도 소진)가 관측되어, 근본원인 ②(프로세스 간 상호배제 부재)가 Windows 파일시스템 레벨의 rename 경합으로 직접 드러남을 확인. Rust 단위테스트(risk2)가 예측한 결함이 실제 바이너리·실제 OS에서도 동일하게 발생함을 교차검증.
- **원인 (파일·함수·라인):**
  - `src-tauri/src/main.rs` · `save_data` (L60–107, 코어는 `sync_core.rs save_data_inner`)
  - ① **버전 = 파일 mtime**, 판정 `server_ts > expected_version + 500`. 500ms 오차 허용 창 안에서 발생한 두 번째 저장은 stale 버전이어도 통과 → lost update. 네트워크 드라이브(SMB)에선 mtime 해상도·클럭 스큐로 창이 더 넓어져 악화.
  - ② `state.write_lock`(Mutex)은 **프로세스 내부 전용**. 사용자마다 별도 인스턴스(별도 AppState) → **프로세스 간 상호배제 없음**. 두 인스턴스의 동시 저장은 직렬화되지 않고 mtime 게이트에만 의존.
  - ③ check(메타데이터 읽기)와 write(rename) 사이에 원자성 없음(TOCTOU).
- **수정 방향 (제안, 미적용):**
  - **(A) 파일 내 단조 증가 리비전 카운터** — payload에 `_rev`를 두고, `save_data`가 저장 직전 현재 파일의 `_rev`를 다시 읽어 `expected_rev`와 비교, 불일치 시 `VERSION_CONFLICT`, 일치 시 `_rev+1`로 기록. mtime·클럭에 무관하게 견고. → 데이터 포맷·클라이언트(App.tsx read/save) 계약 변경 수반(교차 계층).
  - **(B) OS 파일 잠금(advisory lock)** — 공유 파일에 대해 저장 구간 전체를 배타 잠금(`LockFileEx`/`fs2` 크레이트)으로 감싸 프로세스 간 직렬화. mtime 게이트를 보완.
  - **(C) 500ms 허용 창 축소·제거** — 미봉책. mtime 해상도 문제로 false conflict 유발 가능, 근본 해결 아님.
  - **권장:** (A)+(B) 조합. **교차 계층 변경(main.rs + App.tsx + 데이터 포맷)이라 별도 작업으로 진행 권고.**
- **수정 내용 (2026-07-03):** `src-tauri/src/sync_core.rs`(신규 `save_data_inner`/`read_data_inner`/`CriticalSection`) + `main.rs`(위임) + `App.tsx`(rev 계약) + `server.ts`(웹 모드). mtime 폐기 → payload 루트 `_rev` CAS + `{path}.critical.lock` 프로세스 간 임계구역 락. 상세 코드·설계는 [§7.1](#71-결함3-수정--rev-기반-cas--임계구역-락).
- **수정 후 테스트 결과:** L2(`l2_tests`) `risk2_fixed_*` 2건 + 대조군 2건 통과(6/6) — stale rev 저장은 이제 `VERSION_CONFLICT`로 거부되고 A의 데이터가 더 이상 유실되지 않음, 스레드 동시저장도 정확히 한쪽만 성공(직렬화 확인). L3 네이티브·L4 NAS(UNC/매핑드라이브) E2E에서도 실제 바이너리로 VERSION_CONFLICT 발생을 재확인(§7.4·§7.5). 상세는 §7.6.

### 결함 #7 — 파일 감시자가 원자적 rename 저장 이후 변경 알림을 조용히 놓칠 수 있음
- **발견:** 2026-07-03 ("최종적으로 E2E 수행" 요청에 따른 L3/L4 재검증 반복 실행 중, "B가 A의 변경을 전파받지 못함"이 간헐적으로 재현되어 추적)

- **상태:** ✅ **수정 완료 (2026-07-03)** — 상세는 [§8.3](#83-결함7-발견·수정--파일-감시자가-rename-이후-변경을-놓침)
- **요약:** `start_file_watcher`(`main.rs`)가 대상 **파일 자체**를 `notify::Watcher`로 감시했는데, `save_data`는 원자적 쓰기를 위해 `.tmp` 파일을 쓴 뒤 `rename()`으로 덮어쓴다. rename은 파일의 기존 핸들/inode를 새 것으로 교체하는 연산이라, 파일 자체에 바인딩된 OS 레벨 watch가 **최초 rename 이후 조용히 끊길 수 있다** — 이후의 변경은 감시자가 감지하지 못해 다른 사용자에게 전파되지 않는다. 부수적으로, `start_file_watcher`가 재호출(mount·reload·파일 열기)마다 무조건 새 감시 스레드를 스폰해 감시 스레드가 누적되는 문제도 함께 있었다.
- **수정:** ① 파일이 아니라 **부모 디렉터리**를 watch하고 이벤트 대상 경로가 우리가 감시하려는 파일명과 일치할 때만 반응하도록 필터링(디렉터리는 rename으로 사라지지 않으므로 watch가 끊기지 않음 — rename 지속형 파일 감시의 표준 해법). ② `AppState`에 감시 중인 경로 집합을 추적해 같은 경로에 대한 중복 스레드 스폰을 방지.

## 4. 설계 개선안 (결함 #3·#4) — ✅ 2026-07-03 구현 완료

> **2026-07-03 갱신:** 아래 설계안(§4 전체)은 계획대로 구현되었다. 실제 코드·검증 결과는 [§7](#7-결함-3·4-수정--rev-기반-cas--락-통합-2026-07-03)을 참조. 이 섹션은 최초 설계 의도를 보존하기 위해 그대로 남긴다.
>
> #3(mtime 버전 경합 lost update)과 #4(락이 저장을 못 막음)는 **동일한 뿌리** — "공유 파일에 대한 낙관적 동시성 제어가 약하다" — 에서 나온다. 따라서 **하나의 통합 재설계**로 다루는 것이 옳다. 개별 미봉책은 서로 충돌하거나 새 결함을 만든다.

### 공통 근본 원인
1. **버전 식별자로 파일 mtime을 사용** — mtime은 (a) 해상도가 FS마다 다르고(NTFS ~수ms, FAT/일부 SMB 1~2s), (b) 머신 간 클럭 스큐에 취약, (c) `save_data`의 500ms 허용 창이 lost update를 정당화한다.
2. **프로세스 간 상호배제 부재** — `AppState.write_lock`(tokio Mutex)은 한 인스턴스 내부에서만 유효. 사용자마다 별도 프로세스 → 공유 파일에 대한 진짜 잠금이 없다.
3. **락과 저장의 분리** — 아이템 락(`locks/item_*.lock`)은 UI 어드바이저리일 뿐 `save_data` 경로가 참조하지 않음. 게다가 락은 아이템 단위, 저장은 파일 전체 단위라 모델이 불일치.
4. **check→write 사이 TOCTOU** — 메타데이터 확인과 rename 사이에 원자성이 없다.

### 설계 개선안 #3 — 낙관적 동시성 재설계 (위험 2)
- **핵심: 파일 내 단조 증가 리비전 `_rev` 도입 (mtime 폐기)**
  - 데이터 payload 루트에 `_rev: number`를 둔다(초기 0).
  - `save_data(path, data, expected_rev)`: 저장 직전 **현재 파일을 다시 읽어** `current._rev` 확인 →
    - `current._rev !== expected_rev` → `VERSION_CONFLICT`(클라이언트가 병합 후 재시도).
    - 일치 → `data._rev = expected_rev + 1`로 기록.
  - mtime·클럭·500ms 창에 전혀 의존하지 않으므로 SMB/오프라인에서도 견고.
- **프로세스 간 원자성 확보 (택1 또는 병행)**
  - **(B1) OS advisory 파일 잠금** — `fs2`/`fs4` 크레이트의 `lock_exclusive()`로 "읽기(_rev)→검증→쓰기" 구간을 배타 잠금. 크로스 프로세스 직렬화. 단, SMB의 advisory lock 지원은 서버 설정에 따라 불완전할 수 있음(주의).
  - **(B2) 락 파일 기반 임계구역** — `path.lock` 파일을 `create_new`(O_EXCL 유사)로 원자적 생성해 임계구역 진입, 실패 시 짧게 재시도. SMB에서도 `create_new`는 대체로 원자적. 스테일 락은 TTL로 정리.
- **영향 범위**
  - **Rust (`main.rs`/`sync_core.rs`):** `save_data` 시그니처 `expected_version: u64` → `expected_rev: u64`, 내부 로직 교체. `read_data`가 `_rev`를 함께 반환.
  - **데이터 포맷:** payload 루트에 `_rev` 필드 추가. 기존 파일(무 `_rev`)은 로드 시 `_rev=0`으로 간주(마이그레이션 불필요, 하위호환).
  - **프론트 (`App.tsx`):** `lastSaveRef`를 mtime→`_rev`로 교체. `save_data` 호출 시 `expectedVersion`→`expectedRev`. `VERSION_CONFLICT` 수신 시 기존 `executeSmartMerge` 경로 재사용(이미 존재). `shared-file-changed` 감지 로직도 `_rev` 비교로 전환.
  - **웹 모드(server.ts):** 동일 개념을 Express `save`/`load` 경로에도 반영(선택). 웹은 Socket.IO 서버가 단일 중재자라 우선순위 낮음.
- **회귀 리스크**
  - 🔴 저장/로드 경로 전면 수정 → 웹·Tauri **양쪽 저장 흐름 회귀** 가능. 특히 `initCompleteData`/`lastSavedPayload` 동기화 타이밍.
  - 🟠 `_rev` 누락 파일과의 하위호환 처리 누락 시 최초 로드에서 항상 충돌할 위험.
  - 🟠 파일 잠금이 비정상 종료 시 스테일 락으로 남아 저장 불가 → TTL/force-release 필요.
- **권장 적용 순서 (단계적, 각 단계 후 테스트)**
  1. `sync_core.rs`에 `_rev` 기반 `save_data_inner_v2` 추가 + L2 테스트를 "충돌 정상 거부/유실 없음" 단언으로 **전환**(현재는 버그 존재 단언).
  2. 임계구역 잠금(B2 락파일) 추가, 스레드 동시 저장 테스트가 **직렬화·무유실**로 통과하도록.
  3. `main.rs` `save_data`/`read_data`를 v2로 위임(전체 `tauri build` 검증 — dlltool 필요).
  4. `App.tsx`를 `_rev` 계약으로 전환, L1 병합 경로와 통합(충돌 시 `executeSmartMerge`).
  5. 웹 모드(server.ts) 반영(선택).
- **대안**
  - **(대안 1) 콘텐츠 해시 기반** — `_rev` 대신 직전 로드 콘텐츠의 해시를 base로 전송, 서버가 현재 해시와 비교. `_rev`와 효과 유사하나 payload가 커지면 비용↑.
  - **(대안 2) 중앙 중재 서버로 일원화** — 오프라인/공유파일 모델을 버리고 항상 Socket.IO 서버 경유. 오프라인 요구사항과 상충하므로 비권장.
  - **(대안 3) append-only 이벤트 로그 + CRDT** — 근본적으로 견고하나 구현 규모가 큼(장기 과제).

### 설계 개선안 #4 — 락 존중 저장 경로 (위험 3)
- **핵심: #3의 낙관적 동시성으로 흡수하는 것이 1순위**
  - `_rev` + 임계구역 잠금이 들어오면 "락 보유 중 타인이 덮어쓰기"는 **버전 충돌로 자동 차단**되고, 충돌 시 `executeSmartMerge`로 셀 단위 병합되어 유실이 사라진다. 즉 #4의 대부분은 #3 해결로 부수 해결.
- **보완: 저장 시 락 검증(선택)**
  - `save_data`가 저장 대상 파일의 `locks/` 중 **타 사용자 소유의 활성 락**이 있으면 경고/거부하는 옵션. 단 파일 전체 저장 모델과 맞추려면 변경된 아이템 diff를 계산해 해당 아이템 락만 검사해야 하므로, diff 계산이 선행돼야 함.
- **영향 범위**
  - **Rust:** `save_data`가 `get_active_locks` 결과를 참조(간단) 또는 diff 계산(복잡).
  - **프론트:** 락 만료(TTL 15s)와 편집 세션 길이의 불일치 재검토 — 장시간 편집 시 락이 먼저 풀려 무방비가 되는 문제(부수 위험)도 함께 다뤄야 함.
- **회귀 리스크**
  - 🟠 저장 거부를 강하게 걸면 정상 저장까지 막아 UX 저하. "거부"보다 "충돌 병합"이 안전.
- **권장:** #3 재설계에 **종속**시켜 함께 진행. 단독 구현 비권장.

## 6. L4 실 NAS/공유드라이브 환경 — 진행 기록 (2026-07-03)

### 6.1 L4 정의 — 무엇을 검증해야 하는가
L3(네이티브 2-인스턴스, 로컬 경로)와 달리 L4는 **"공유 파일이 실제로 네트워크 위에 있을 때"** 만 드러나는 계층을 검증한다. 구체적으로:
1. **UNC 경로 변환** — 사용자가 매핑 드라이브 문자(`Z:\...`)로 경로를 지정해도, `convert_to_unc_path`(`main.rs` L423-499)가 실제 `\\서버\공유\...` UNC 형태로 정확히 변환하는가(WMI `ProviderName` 조회 → `net use` 파싱 → `canonicalize` 폴백의 3단 로직).
2. **네트워크 경로 위에서의 파일 잠금(`locks/`)** — `acquire_item_lock`/`release_item_lock`/`get_active_locks`가 UNC 경로에서도 파일 기반으로 정상 동작하는가.
3. **원자적 저장(`save_data`)** — tmp write + rename이 네트워크 경로에서도 원자성을 유지하는가, mtime 버전 게이트(결함#3)가 네트워크 지연/해상도 차이로 더 자주 실패하지 않는가.
4. **감시자 전파(`notify` watcher)** — `start_file_watcher`의 `ReadDirectoryChangesW` 기반 감시가 네트워크 공유에서도 변경을 감지해 `shared-file-changed`를 발생시키는가(일부 SMB/NAS 구현은 change-notify를 지원하지 않거나 지연시킬 수 있음).
5. **오프라인/재연결** — 공유 드라이브 연결이 끊겼다가 복구될 때 앱이 어떻게 동작하는가(에러 처리, 재시도, 사용자 알림).
6. **실 NAS 고유 요인** — 진짜 네트워크 지연·클럭 스큐·이기종 SMB 서버(제조사별 구현 차이)·동시 접속자 수 증가 시의 동작. 이 부분은 로컬 시뮬레이션으로는 재현 불가능.

### 6.2 자동 재현 가능성 조사 — admin 권한 없이 "네트워크 공유" 흉내내기
| 방법 | admin 필요? | 결과 |
|---|---|---|
| `net share` (신규 SMB 공유 생성) | ✅ 필요 | ❌ 시도 결과 `Access is denied`(System error 5) 확인. 이 머신에서 비승격 상태로는 불가. |
| `subst`(드라이브 문자 ↔ 로컬 폴더 별칭) | ❌ 불필요 | ✅ 가능. 단, **네트워크 드라이브가 아니므로** `convert_to_unc_path`의 WMI 분기를 타지 않음(아래 6.4 결과) — UNC 변환 로직 자체 검증에는 부적합, 대신 "네트워크 아닌 드라이브 오탐 방지" 회귀 검증에 사용. |
| `\\localhost\C$\...`(Windows 기본 관리자 공유, loopback) | ❌ **불필요** — 신규 생성이 아니라 **이미 존재하는** 관리자 공유(C$)에 접근만 하는 것이라 `Test-Path \\localhost\C$\...` 즉시 `True` 확인. 현재 세션이 비승격(`whoami /groups`상 Administrators 그룹이 "Group used for deny only")임에도 loopback 접근은 허용됨. | ✅ **채택.** 진짜 SMB 프로토콜 스택(Windows Server 서비스 + CIFS/SMB 리다이렉터)을 경유하는 UNC 경로를 admin/신규 공유 없이 확보 가능. |
| `net use <드라이브>: \\localhost\C$\...`(위 관리자 공유를 드라이브 문자에 매핑) | ❌ 불필요(기존 공유에 연결만) | ✅ 가능. 단, **드라이브 매핑은 로그온 세션에 종속되며 이 자동화 환경에서는 셸 호출 간 지속되지 않음**(다음 PowerShell 호출에서 `net use` 목록이 비어 있음 확인) → 매핑 생성·테스트·해제를 **하나의 셸 호출 안에서 연속 실행**하는 방식으로 우회. |

**결론:** `\\localhost\C$\...`(관리자 공유 loopback)를 "네트워크 공유"로 채택. 진짜 SMB 스택을 타므로 로컬 NTFS 직접 접근과는 다른 코드 경로(리다이렉터)를 실제로 거치는 유효한 시뮬레이션이며, `net use`로 드라이브 문자 매핑까지 재현해 `convert_to_unc_path`의 WMI 조회 분기까지 검증 가능. 다만 **loopback이므로 실 네트워크 지연·클럭 스큐·이기종 NAS 벤더 차이는 재현 불가**(§6.6 수동 체크리스트로 이관).

### 6.6 자동 검증 결론
- UNC 경로 변환(드라이브 문자 → WMI 조회 → UNC), 네트워크 경로 위의 파일 잠금(`locks/`), 원자적 저장, 파일 감시자 기반 협업 전파, mtime 버전 게이트 충돌(결함#3) 재현 — **모두 loopback UNC/매핑드라이브 환경에서 정상 동작 확인, 로컬(L3) 대비 신규 결함 없음.**
- 충돌(lost update) 동작은 로컬 경로와 동일한 빈도·양상으로 재현되어, loopback 환경에서는 결함#3이 "네트워크라서 더 악화"되는 징후는 관측되지 않았다(단, loopback은 지연이 거의 0에 가까워 **진짜 원격 NAS의 지연·클럭 스큐를 대변하지 못한다** — §4 근본원인 분석에서 우려한 "SMB에서 창이 더 넓어져 악화"는 여전히 이론적으로 유효하며, 실 NAS 없이는 확정할 수 없다. 아래 수동 체크리스트 항목 4 참조).

### 6.7 수동 체크리스트 — 실 NAS/2번째 물리 PC가 있어야 하는 부분
아래는 이 세션에서 자동화하지 않은(할 수 없는) 항목이다. 실 NAS 장비 또는 두 번째 물리 PC 확보 시 아래 순서로 수행 권장:

1. **실 NAS 공유경로 지정**
   - [ ] NAS(Synology/QNAP/Windows Server 등)에 SMB 공유 생성, 두 PC 모두에서 매핑 드라이브 또는 UNC로 접근 가능한지 확인.
   - [ ] 설정화면에서 해당 UNC/매핑 경로를 `activeDataPath`로 지정, "설정 저장" → `convert_to_unc_path`가 실제 NAS의 UNC 루트로 정확히 변환하는지 확인(§6.4 방식과 동일하되 진짜 원격 서버 대상).
2. **두 물리 PC 동시 편집**
   - [ ] PC1·PC2에서 각각 앱 실행, 같은 NAS 공유파일을 가리키도록 설정.
   - [ ] PC1에서 요구항목 수정 → PC2에서 파일감시(`notify`)로 몇 초 내 전파되는지 실측(로컬/loopback보다 지연이 클 것으로 예상 — 몇 초까지 정상 범위인지 기준 수립 필요).
   - [ ] PC1·PC2가 동시에 같은 셀/필드를 편집 → 병합(결함#1·#2 수정 반영) 또는 충돌 처리(결함#3, lost update)가 §5.8/§6.4와 동일하게 일어나는지, 아니면 실 네트워크 지연으로 다른 양상(예: 더 잦은 VERSION_CONFLICT, 더 큰 데이터 유실 창)이 나타나는지 확인.
3. **아이템 락 실사용 시나리오**
   - [ ] PC1이 특정 요구항목을 편집 시작(락 획득, TTL 15초) → PC2가 같은 항목 편집 시도 시 잠금 경고가 뜨는지, TTL 만료 후 자동 해제되는지 확인.

## 7. 결함 #3·#4 수정 — `_rev` 기반 CAS + 락 통합 (2026-07-03)

> §4 설계 개선안을 그대로 구현했다. 목표: (1) mtime 버전 게이트를 파일 payload 루트의 단조 증가 `_rev` 카운터 기반 compare-and-swap(CAS)으로 교체, (2) read→비교→write 임계구역을 프로세스 간에도 유효한 파일 잠금으로 보호, (3) 저장 시 아이템 락과 실제로 연동. 웹(Express)·네이티브(Tauri) 양쪽 저장 경로 모두 반영. L1~L4 전 계층 재검증으로 회귀 없음 확인.

### 7.1 결함#3 수정 — `_rev` 기반 CAS + 임계구역 락
- **파일:** `src-tauri/src/sync_core.rs` (전면 재작성)
- **`_rev` 계약:** payload 루트에 `_rev: number`(단조 증가, 초기 0). `_rev` 필드가 없는 기존 파일은 읽을 때 자동으로 0으로 간주(마이그레이션 불필요, 하위호환).
- **`save_data_inner(path, data, expected_rev, user_id)`:**
  1. `CriticalSection::acquire(path)` — `{path}.critical.lock`을 `OpenOptions::create_new`(O_EXCL)로 생성해 임계구역 진입. 실패 시 50ms 간격으로 최대 5초 재시도. 락 파일이 5초 이상 방치되면(보유 프로세스 비정상 종료로 간주) 스테일 락으로 판단해 강제 회수. `Drop` 구현으로 함수 종료 시(성공·에러·패닉 무관) 자동 해제.
  2. 임계구역 안에서 현재 파일을 다시 읽어 `_rev` 확인 → `current_rev != expected_rev`면 `VERSION_CONFLICT`.
  3. (결함#4 연동, §7.2) 아이템 락 충돌 검사.
  4. `_rev = expected_rev + 1`로 갱신해 원자적 쓰기(tmp write → 기존 파일 `.bak` 백업 → rename, 기존 재시도 정책 유지).
- **`read_data_inner(path)`:** 기존 `.bak` 자동 복구 로직을 유지하며 `(parsed_value, rev)` 튜플을 반환.
- **mtime·500ms 오차창을 완전히 폐기** — 더 이상 시계·파일시스템 해상도에 의존하지 않으므로 SMB/네트워크 지연에도 견고(L4 §7.5에서 실증).
- **프로세스 간 상호배제:** 기존 `AppState.write_lock`(tokio Mutex, 프로세스 내부 전용)은 그대로 두고, `CriticalSection`(파일 기반)을 추가해 **다른 프로세스(다른 사용자 인스턴스)** 간에도 진짜 상호배제를 확보(§4 근본원인 ② 해결).

### 7.2 결함#4 수정 — 저장 시 아이템 락 diff 검증
- **파일:** `src-tauri/src/sync_core.rs`
- `extract_requirement_map(value)`: payload의 `tabDataMap.*.requirements[]`에서 `id → 정규화된 내용 문자열` 맵 추출.
- `find_locked_conflict(locks_dir, old, new, user_id)`: 저장 직전 파일(old)과 저장하려는 내용(new)을 비교해 **변경되거나 삭제된** 요구항목 id를 뽑고, 그 id에 대한 **활성** 아이템 락이 있고 **소유자가 저장자(user_id)와 다르면** 해당 id를 반환.
- `save_data_inner`가 이 함수의 결과가 `Some(id)`면 `ITEM_LOCKED:<id>`로 저장을 거부. 락 소유자 본인의 저장, 락과 무관한 다른 항목만 바뀐 저장은 영향받지 않음(L2 `risk3_fixed_item_lock_blocks_conflicting_save`의 3개 서브케이스로 검증).
- **파일 전체 저장 ↔ 아이템 단위 락의 모델 불일치(§4 근본원인 ③)를 diff 계산으로 해소** — "무엇이 바뀌었는지" 알 수 없다는 기존 한계를 payload 비교로 극복.

### 7.6 mergeEngine.ts 탭 레벨 필드 병합 정책에 대한 정확한 설명
`computeSmartMerge`(`src/lib/mergeEngine.ts`)는 **요구항목(`requirements[]`) 필드**(title/priority/status/dueDate/customColumns/assignees)에 대해서만 3-way 충돌 검출(`conflicts.push(...)`)을 수행한다. **탭 레벨 메타데이터**(`dashboardTitle`/`dashboardDesc` 등)는 다른 정책을 따른다(L58-79): theirs를 먼저 적용한 뒤, "내가 base 대비 바꿨으면" 무조건 내 값으로 덮어쓴다 — **충돌 검출이 아예 없다.** 이는 `_rev`/락 수정과 무관한, `mergeEngine.ts`의 기존(수정 범위 밖) 설계다(파일 상단 주석: "동작은 원본과 100% 동일하게 유지한다"). L3/L4의 충돌 시나리오가 `dashboardTitle`(탭 필드)을 사용하므로 "한쪽만 반영"이 관찰되는 것은 이 정책 때문이며, 결함#3 수정이 막으려던 것(레이스에 의한 무작위·무음 유실)과는 다른, 결정적이고 의도된 동작이다. 요구항목 필드의 동시 편집은 L1에서 이미 "충돌로 검출된다"가 검증되어 있다(그룹 A 2번째 케이스). **후속 과제 후보:** 탭 레벨 필드도 요구항목처럼 충돌 검출 대상에 포함할지는 별도 제품 결정 사항으로 남긴다(이번 세션 범위 밖).

### 7.7 결함 #6 — 충돌 모달의 "내 값 강제 저장" 옵션이 사실상 항상 실패함
- **발견:** 2026-07-03 (`_rev` 마이그레이션 중 코드 검토로 발견)
- **심각도:** 🟠 Medium (충돌 발생 시 사용자가 "내 값 유지"를 선택해도 조용히 아무 일도 일어나지 않음 — 명시적 에러 표시조차 없음)
- **상태:** ✅ **수정 완료 (2026-07-03)**
- **재현 방법(기존 코드 분석, 수정 전 mtime 체계 기준):**
  - `ConflictModal`의 "mine"(강제 덮어쓰기) 전략은 `invoke("save_data", {..., expectedVersion: 0})`을 호출 — "버전 무시하고 강제 저장"을 의도한 매직 넘버.
  - 그러나 옛 mtime 게이트는 `server_ts > expected_version + 500`이며, 충돌이 발생했다는 것은 곧 대상 파일이 **이미 존재**한다는 뜻이므로 `server_ts`(실제 mtime, 초대형 epoch-ms 값)가 `0 + 500`보다 항상 크다 → **매번 `VERSION_CONFLICT`로 실패.**
  - 이 호출에는 **`try/catch`가 없어** 실패가 처리되지 않은 프라미스 거부로 조용히 사라짐(콘솔에도 안 남을 수 있음). 사용자는 "내 값으로 저장"을 눌렀다고 생각하지만 실제로는 아무 것도 저장되지 않음.
- **원인 (파일·함수·라인):** `src/App.tsx` `ConflictModal.onResolve`의 `strategy === "mine"` 분기(수정 전 기준 L1646-1654 부근).
- **수정 내용 (2026-07-03):** 저장 직전 **현재 파일을 다시 읽어 최신 `rev`를 확인**하고 그 `rev`로 저장하도록 변경(진짜 "마지막에 확인한 상태 기준 강제 승리"). `try/catch`를 추가해 실패 시에도 콘솔에 로그가 남도록 함.
  ```ts
  if (strategy === "mine") {
    if (dataPayload === lastSavedPayload.current) return;
    const { invoke } = await import("@tauri-apps/api/core");
    try {
      const current: any = await invoke("read_data", { path: dbPath });
      const newModified: any = await invoke("save_data", {
        path: dbPath, data: dataPayload,
        expectedRev: current.rev ?? 0, userId: currentUser.id,
      });
      lastSaveRef.current = Number(newModified);
      lastSavedPayload.current = dataPayload;
    } catch (e) {
      console.error("Force overwrite save failed:", e);
    }
  }
  ```
- **수정 후 테스트 결과:** 별도의 전용 자동화 케이스는 작성하지 않았음(수동 코드 리뷰로 원인 확정 + 수정, `save_data`/`read_data` 자체의 `_rev` CAS 정합성은 L2에서 충분히 검증됨). **후속 검증 권장:** 실제 충돌 모달을 띄워 "내 값 유지" 버튼을 눌러 저장이 성공하는지 수동 확인(§6.7 수동 체크리스트에 추가 권고).

### 8.3 결함#7 발견·수정 — 파일 감시자가 rename 이후 변경을 놓침
- **발견 경위:** L3(`collab.mjs`) 재실행 중 "B가 A 변경을 전파받음" 단계가 간헐적으로 실패(B의 대시보드 제목이 A의 편집을 반영하지 못하고 기본값에 머묾)하는 것을 2회 재현. 매번 콘솔에 동일한 `[TAURI] Couldn't find callback id <N>` 경고가 대량으로 반복 출력되는 것을 단서로 원인을 추적.
- **원인:**
  1. `start_file_watcher`(`main.rs`)가 대상 **파일 자체**를 `watcher.watch(Path::new(&path), NonRecursive)`로 감시하고 있었다. 그런데 `save_data`(§7.1)는 원자적 쓰기를 위해 `.tmp` 파일에 쓴 뒤 `fs::rename(tmp, path)`로 **덮어쓴다.** rename은 OS 레벨에서 기존 파일의 핸들/디렉터리 엔트리를 새 파일로 교체하는 연산이므로, **파일 자체에 바인딩된 watch가 최초 rename 이후 조용히 끊길 수 있다**(플랫폼/파일시스템에 따라 발생 여부가 다름 — 바로 이 때문에 간헐적으로만 재현됨). watch가 끊긴 이후의 모든 변경은 감시 스레드가 감지하지 못한다.
  2. `start_file_watcher`는 `fetchData()`가 mount·reload·파일 열기마다 무조건 재호출하는데, 매 호출마다 새 감시 스레드+`notify::Watcher` 인스턴스를 무조건 스폰해 **감시 스레드가 누적**되고 있었다(추적/정리 메커니즘 없음). 같은 파일에 대해 여러 OS 레벨 watch 핸들이 경합하면서 위 ①의 증상을 더 자주 유발한 것으로 보인다.
- **수정 (`src-tauri/src/main.rs`):**
  - `AppState`에 `watched_paths: std::sync::Mutex<HashSet<String>>` 추가. `start_file_watcher`는 이미 감시 중인 경로면 새 스레드를 스폰하지 않고 즉시 성공 반환.
  - 감시 대상을 **파일이 아니라 부모 디렉터리**로 변경(`target_path.parent()`)하고, `notify::EventKind::Modify`/`Create` 이벤트의 `paths` 중 우리가 감시하려는 **파일명과 일치하는 것이 있을 때만** 해시 비교·`emit`을 수행하도록 필터링. 디렉터리 자체는 rename으로 사라지지 않으므로 watch가 끊기지 않는다(rename 지속형 파일 감시의 표준 해법).
- **수정 후 재검증:**
  - 재빌드(`npm run build` + `npx tauri build --target x86_64-pc-windows-gnu --no-bundle`) 후 L3(`collab.mjs`) **연속 2회 9/9 통과** — 두 회차 모두 "B가 A 변경을 전파받음" 정상 확인.
  - L4(`collab_l4.mjs`) UNC 경로 **14/14 통과**, 매핑 드라이브 경로 **14/14 통과**(1회 재시도 후) — 전파 실패 재현되지 않음.
- **테스트 스크립트 견고성 개선(부수):** `collab.mjs`/`collab_l4.mjs`의 결함#4(ITEM_LOCKED) 검증 단계가, 배경에서 진행 중인 자동저장·병합 재시도로 `_rev`가 계속 움직이는 동안 무관한 `VERSION_CONFLICT`로 오탐하는 문제를 발견 → 저장 시도 직전마다 `_rev`를 새로 읽고 우연한 `VERSION_CONFLICT`는 최대 5회 재시도하도록 하드닝(결함이 아니라 테스트 스크립트의 타이밍 가정 문제였음, §8.4 참조).

### 8.5 관찰 사항 갱신 — 트랜지언트 렌더 오류와 시스템 부하의 상관관계

> **[2026-07-04 종결]** 이 관찰사항은 §11에서 최종 조사되어 **결함 #9로 확정·수정 완료**되었다. 근본 원인은 시스템 부하가 아니라 "부분 페이로드 → 병합이 tabs:[]로 오염 → 오염 파일 로드 시 확정 크래시"였다(아래 부하 상관관계 추정은 부차 요인). 상세 증거 사슬은 §11.1 참조.
§7.5에서 원인 미확정으로 남겼던 `TypeError: Cannot read properties of undefined (reading 'dashboardTitle')` 렌더 오류가 이번 최종 검증 중 매핑 드라이브 시나리오에서 **1회는 `ErrorBoundary`가 흡수하지 못할 정도로 심하게 재현**되어(`#sidebar-container` 대기 자체가 타임아웃) 테스트가 완전히 실패했다. 실패 직후 시스템을 점검한 결과 하루 동안의 반복 테스트로 **`msedgewebview2.exe` 고아 프로세스가 18개까지 누적**되어 있었음을 확인(각 E2E 실행이 인스턴스당 다수의 WebView2 헬퍼 프로세스를 남기고, `killAll()`이 메인 `.exe`만 종료하고 하위 WebView2 프로세스는 정리하지 못함). 고아 프로세스를 정리한 뒤 동일 시나리오를 재시도하자 **정상 통과(14/14)**했고, 오류가 1회 재현됐으나 이번엔 `ErrorBoundary`가 정상적으로 흡수해 이후 모든 단계가 통과했다. **잠정 결론:** 이 렌더 오류는 실제 제품 결함이라기보다는(mergeEngine·rev·락 로직과는 무관), 반복 테스트로 시스템 자원이 소모될수록 온보딩 리로드 시퀀스의 타이밍이 불안정해지며 드러나는 현상으로 보인다. 다만 근본 원인(정확히 어떤 렌더 시점에 `tabs`가 비거나 `activeTabInfo`가 undefined가 되는지)은 여전히 확정하지 못했으므로 결함 번호는 부여하지 않고, §6.7 수동 체크리스트에 "장시간 세션·많은 재로드 후 이상 렌더 발생 여부"를 관찰 항목으로 유지한다. **후속 조치 제안(미착수):** `e2e-native/*.mjs`의 `killAll()`이 `business-requirements.exe`뿐 아니라 그 자식 WebView2 프로세스도 정리하도록 개선하면(예: WMI로 부모 PID 기준 자식 프로세스 종료) 테스트 세션 간 자원 누적을 막아 이런 상관관계를 근본적으로 줄일 수 있을 것으로 예상.

### 9.1 🔴 신규 결함 발견 — 결함 #8 (분석 중 실증)

**결함 #8 — 아이템 락 ID에 콜론(`:`) 포함 → Windows/NAS에서 락 시스템이 사실상 무력화됨**
- **발견:** 2026-07-04 (본 분석 중 코드 검토 → Node.js 실증 테스트로 확정)
- **심각도:** 🔴 High — 실사용 UI의 락 기능이 세 갈래로 동시에 깨져 있음
- **상태:** 📐 **확정 · 수정 대기** (분석 요청 범위라 수정은 승인 후 진행)
- **원인:** UI(`Spreadsheet.tsx` L553/576/611)가 락 itemId로 `` `${activeTabId}:${rowId}` ``(예: `requirements:REQ-001`)를 사용 → 락 파일명이 `item_requirements:REQ-001.lock`이 됨. **콜론은 Windows 파일명 금지 문자**(NTFS ADS 구분자).
- **실증 결과(NTFS, Node.js로 재현):**
  ```
  1) 쓰기: 성공(예외 없음) — 그러나 실제로는 'item_requirements' 파일의 ADS(대체 데이터 스트림)로 생성됨
  2) 같은 경로 exists 체크: true       ← 같은 클라이언트의 재획득/충돌검사는 우연히 동작
  3) 디렉터리 목록: 'item_requirements' (size=0) 하나뿐 — .lock 파일이 안 보임
  4) 'item_*.lock' 패턴 매치: 0건      ← get_active_locks가 아무것도 못 봄
  ```
- **세 갈래 파급:**
  1. **락 가시성 상실(NTFS):** `get_active_locks`(read_dir 기반)가 UI 락을 전혀 못 보므로, 2초 폴링으로 표시되는 "OO님이 편집 중" 인디케이터(Spreadsheet.tsx L2815-2820)가 **한 번도 표시되지 않음**. 다른 사용자가 같은 셀을 열 때의 직접 경로 충돌 거부만 우연히 동작(같은 ADS 경로 exists 체크).
  2. **결함#4 수정 미작동(실 UI 경로):** `find_locked_conflict`는 `get_active_locks_from_dir`(디렉터리 스캔)로 락을 조회하므로 ADS 락을 못 봄. **게다가 키도 불일치** — 저장 검증은 요구항목의 `r.id`(`REQ-001`)로 조회하는데 UI 락 키는 `requirements:REQ-001`. 즉 디렉터리 스캔이 됐더라도 매치가 안 됨. §7~§8의 L2/L3/L4 검증이 통과한 것은 테스트가 콜론 없는 평면 ID(`REQ-LOCK-TEST`)를 사용했기 때문 — **테스트 ID 형식과 실 UI ID 형식의 불일치가 이 결함을 은폐**했다.
  3. **실 NAS(Samba/Linux) 치명 오작동 가능:** SMB 프로토콜/Samba는 콜론 포함 파일명을 거부하는 것이 일반적 → `acquire_item_lock`의 `fs::write`가 Err → UI는 이를 락 충돌로 해석해 **"다른 사용자가 편집 중" 경고를 띄우고 편집 자체를 차단**(L580-588). 즉 실제 NAS에서는 아무도 편집을 못 하게 될 수 있음.
  4. (부수) TTL 만료 정리 로직도 ADS 컨테이너 파일(`item_requirements`, `.lock` 접미사 없음)을 못 지워 탭 수만큼 쓰레기 파일이 누적.
- **수정 방향(제안):** ① itemId를 파일명에 쓰기 전 새니타이즈(금지문자 치환/인코딩) — Rust 쪽 락 함수 3종에서 일괄 처리해야 우회 경로가 없음. ② 락 키 네임스페이스 정합: 저장 시 검증(`find_locked_conflict`)이 `tabId:rowId` 형식을 이해하도록 통일. ③ L2에 콜론 포함 ID 회귀 테스트 추가.

## 10. P0·P1 안정화 수정 (2026-07-04, `feat/multiuser-stability`)

> §9 분석에서 도출한 P0(결함#8)과 P1(신뢰성 4건)을 구현하고 전 계층 재검증했다. P2·P3은 후속 과제로 남긴다.

### 10.1 P0 — 결함 #8 수정: 락 ID 인코딩 + 네임스페이스 정합
- **`sync_core.rs`:**
  - `encode_lock_id`/`decode_lock_id` 신규 — Windows 금지 문자(`: / \ * ? " < > |`, 제어문자)와 `%` 자체를 `%XX`(hex)로 치환하는 전단사 인코딩. `acquire`/`release`/`get_active_locks` 3개 락 함수가 파일명 생성·해석 시 일괄 사용하므로 우회 경로 없음. 스캔 시 디코딩해 **원래 itemId(콜론 포함)를 그대로 키로 반환** → UI 인디케이터(`activeLocks["탭ID:행ID"]`) 조회가 즉시 복원됨.
  - `extract_requirement_map` 키를 `행ID` → `"탭ID:행ID"`로 변경(UI 락 키와 정합). `find_locked_conflict`는 1차로 정확 매치, 2차로 평면 행ID 폴백(레거시/외부 락 방어). `ITEM_LOCKED:` 에러도 네임스페이스 포함 형태(`ITEM_LOCKED:탭ID:행ID`)로 반환.
- **검증:** L2 신규 테스트 `defect8_fixed_colon_item_ids_work_end_to_end` — ① 인코딩 왕복 무손실 + 금지문자 부재, ② 콜론 락이 디렉터리 스캔에 진짜 파일(`item_…%3A….lock`)로 1건 검출(ADS 아님), ③ `get_active_locks`가 원래 키로 반환, ④ 타 사용자 저장 `ITEM_LOCKED:탭ID:행ID` 거부, ⑤ 해제 후 정상 저장. **통과.**

### 10.2 P1-R1 — `.bak` 복구 조건 강화 (최신 데이터 롤백 방지)
- **`sync_core.rs::read_data_inner` 재작성:**
  - 일시 IO 오류(SMB 공유 위반 등) → 100ms×3 재시도 → 소진 시 **`.bak`를 건드리지 않고** `READ_RETRY_EXHAUSTED` 에러 반환(호출부는 기존 상태 유지).
  - `NotFound`(크래시로 파일 소실) 또는 읽기는 성공했으나 JSON 파싱 실패(손상) → 기존대로 `.bak` 복구(정당한 복구 경로 유지).
- **검증:** L2 신규 2건 — `r1_fixed_corrupt_file_recovers_from_bak`(손상→복구 정상), `r1_fixed_transient_lock_does_not_rollback_to_bak`(Windows `share_mode(0)`로 배타 점유를 흉내 → 에러 반환 확인 + 본 파일 rev5가 구버전 bak(rev1)로 **롤백되지 않음** 확인). **통과.**

### 10.3 P1-R2 — 락 TTL 판정을 파일 mtime(파일서버 시계) 기준으로 전환
- **`sync_core.rs`:** `lock_age_secs()` 신규 — 락 파일 **mtime의 경과 시간**으로 나이 판정. 기존 `acquiredAt`(획득자 로컬 시계) vs `now`(판정자 로컬 시계)의 쌍별 스큐(10명이면 조합 폭발)를, 모든 클라이언트가 **NAS 시계 하나만 상대**하는 구조로 축소. 하트비트(5초)가 mtime을 갱신하므로 TTL(15초) 대비 10초의 스큐 여유. mtime이 미래(클라이언트가 서버보다 느림)면 나이 0 = 락 유지(조기 만료 방지, 안전한 방향). `acquiredAt`은 표시/디버그용으로 유지.
- **검증:** L2 신규 `r2_fixed_ttl_uses_file_mtime_not_writer_clock` — `acquiredAt`을 1시간 과거로 조작한 락 파일(구식 판정이면 즉시 만료로 오판)이 mtime 기준으로는 활성으로 판정되고 타 사용자 획득도 거부됨. **통과.**

### 10.4 P1-T2/R5 — App.tsx 충돌 백오프 + 보류 저장 자동 재시도
- **`App.tsx` 자동저장 이펙트:**
  - `VERSION_CONFLICT` → 즉시 재시도 대신 **지수 백오프+지터**(250ms×2ⁿ 최대 3초 + 0~250ms 랜덤) 후 `executeSmartMerge` — 다중 사용자 동시 충돌 시 재시도 시점을 분산(thundering herd 방지). 저장 성공 시 스트릭 리셋.
  - `ITEM_LOCKED` → 안내 후 **3초 후 자동 재시도**(`saveRetryNonce`로 이펙트 재점화 — 기존에는 다음 상태 변경까지 무기한 보류될 수 있었음). 락 TTL 최대 15초이므로 수 회 내 해소.
  - `LOCK_TIMEOUT` → "오프라인 모드" 오표시 대신 **"공유 파일 사용량이 많아 저장 대기 중"** 으로 구분 표시 + 1.5~3초 지터 재시도.
- **잔여 리스크(문서화):** 보류 직후 수 초 내 앱을 종료하면 해당 변경이 서버에 못 미침(localStorage에는 남음). 완전한 보호(재시작 시 병합)는 P2 후속.

## 11. 결함 #9 조사·수정 — §8.5 관찰사항 종결 (2026-07-04)

> §8.5에서 "원인 미확정, 결함 번호 미부여"로 남겼던 트랜지언트 렌더 오류(`dashboardTitle` TypeError)를 최종 조사했다. 결론: **무해한 타이밍 이슈가 아니라 실제 결함(#9)** — 부분 페이로드가 병합에서 "전체 삭제"로 오해석되어 공유 파일이 `tabs: []`로 영구 오염되고, 오염된 파일은 로드 즉시 확정 크래시를 일으킨다. §8.5의 "시스템 부하 상관관계" 추정은 부차 요인이었다.

### 11.1 조사 방법과 증거 사슬
1. **코드 추적:** 크래시 지점은 App.tsx 렌더의 `activeTabInfo.dashboardTitle` — `activeTabInfo = tabs.find(...) || tabs[0]`이므로 `tabs === []`일 때만 undefined가 될 수 있음. 즉 "누가 tabs를 빈 배열로 만들었는가"가 핵심 질문.
2. **파일 포렌식(결정적 증거):** 이전 L4 실행들이 남긴 데이터 파일 3개를 검사 —
   ```
   workspace_l4_final3.json      → tabs: []  (_rev 20)
   workspace_l4_d8.json          → tabs: []  (_rev 11)
   workspace_l4_mapped_final2.json → tabs: []  (_rev 35)
   ```
   **세 파일 모두 `tabs: []`로 오염**되어 있었다. tabDataMap에는 4개 탭 데이터가 남아 있으므로 "탭 배열만" 사라진 것 — 병합 삭제 필터의 서명과 일치.
3. **오염 경로 확정:** E2E의 결함#4 검증 단계가 `{tabDataMap:{t_locktest:...}}`(tabs 키 없음) 부분 페이로드를 direct-invoke로 저장 → 상대 인스턴스의 `executeSmartMerge` → `computeSmartMerge`의 삭제 필터가 `(theirData.tabs || [])` 폴백으로 **"상대가 모든 탭 삭제"로 오판** → `tabs: []` 병합 결과를 자동 저장 → 파일 영구 오염.
4. **크래시 실증(수정 전 빌드, `verify_defect9.mjs`):** 오염된 `workspace_l4_d8.json`을 로드하는 단일 인스턴스 프로브 — 콘솔 스택 캡처:
   ```
   TypeError: Cannot read properties of undefined (reading 'dashboardTitle')
       at http://tauri.localhost/:543:3364  (이하 React 커밋 스택)
   화면: "예상치 못한 오류가 발생했습니다 … 앱 다시 시작"  ← ErrorBoundary 전체 화면 폴백
   ```
   **결정적 재현 — "트랜지언트"가 아니라 오염 파일에서는 매 실행 확정 크래시.** E2E에서 일시적으로 보였던 것은 스크립트가 곧 파일을 초기화/덮어썼기 때문.
5. **§8.5 재현 패턴 정합:** L4에서만 관측된 이유 — L4는 매 실행 **새** 파일을 쓰지만 머신 전역 Rust 설정은 온보딩 완료 전까지 **직전 실행의 오염 파일**을 가리킴. L3는 실행 시작 시 같은 파일을 `{}`로 리셋해 오염이 소거됨. 관측 시점(온보딩 단계)·간헐성(직전 실행의 마지막 저장이 부분 페이로드였는지 여부)까지 전부 설명됨.

아래는 loopback 시뮬레이션으로 대체 불가능한, **실 NAS 장비 + 2대 이상의 물리 PC**가 필요한 항목이다:

## 14. P2·P3 최적화 (2026-07-04, `feat/multiuser-stability`)

> §9.5/§10.7에서 후속 과제로 남긴 P2(전파·부하)·P3(성능) 항목을 구현했다. 각 항목마다 근거·측정·회귀 검증을 남기고, 위험/대규모 리팩터가 필요한 항목은 권고안으로 기록한다.

### 14.2 P3-2 — computeSmartMerge O(n²)→O(n) (실측)
- **수정:** `src/lib/mergeEngine.ts` — 요구항목 조회를 `findIndex`/`includes`(선형)에서 `Map`/`Set`으로 교체. 재번호용 최대 ID도 충돌마다 전체 reduce하던 것을 1회 계산+증분 갱신으로. 동작 동일(L1 14케이스 가드).
- **측정:** `tests/perf/mergeBench.ts`(신규, 재실행 가능) — 요구항목 N건, 양쪽이 각각 10% 편집+소수 추가/삭제, 5회 중앙값:

| N | Before | After | 개선 |
|---|---|---|---|
| 500 | 4.5 ms | 1.2 ms | 3.8× |
| 1,000 | 15.4 ms | 1.5 ms | 10× |
| 3,000 | 137.5 ms | 4.7 ms | 29× |
| 6,000 | 364.9 ms | 7.0 ms | 52× |
| 10,000 | **1,197.6 ms** | **14.5 ms** | **83×** |

- Before는 명확한 O(n²) 곡선(메인 스레드 1.2초 정지 = UI 프리즈), After는 준선형. 10인 환경에서 병합은 초당 수 회 실행될 수 있으므로(§9.2 T3) 체감 효과가 크다.

### 14.3 T3+IO — 병합 코얼레싱 + 사전판독 재사용 (`App.tsx`)
- `requestSmartMerge(path, preRead?)` 신규: 400ms 코얼레싱 창 + 진행 중 가드(pending 1회 재실행) — 감시자/폴링이 연속으로 알릴 때 병합을 1회로 묶는다. 모든 병합 트리거 5곳이 이 경유로 통일(직접 호출 제거).
- `executeSmartMerge(path, preRead?)`: 감시자/폴링이 방금 read_data한 응답을 재사용 → **원격 변경 1건당 NAS 읽기 2회→1회(50% 절감, 설계 확정치)**. stale preRead는 _rev CAS가 방어(최악 VERSION_CONFLICT 1회 후 재병합 수렴).

### 14.4 R6 — 폴링 폴백 하이브리드 (`App.tsx`)
- 기존 2초 락 폴링 인터벌에 **mtime 힌트 검사**(메타데이터 조회 1회, 본문 미판독)를 추가. mtime이 움직였을 때만 read_data→rev 비교→병합 트리거. 감시자가 침묵하는 NAS에서도 전파가 최대 ~2.4초(폴링 2s+코얼레싱 0.4s) 내 보장. 정상 경로 추가 비용은 2초당 메타데이터 1회.
- mtime은 "힌트"로만 사용(결함#3 교훈 — 판정은 언제나 `_rev`).

### 14.5 R3 — 스테일 임계락 원자 회수 (`sync_core.rs`)
- remove 기반 회수를 **rename 선점**으로 교체: 스테일 판정 → `rename(lock, lock.claim.{pid})`(원자) → 선점한 파일 mtime 재검증 → 스테일이면 삭제, 그 사이 교체된 신선한 락이었으면 원상 복구. "판정과 삭제 사이에 교체된 살아있는 락을 지우는" 경로가 사라짐.
- **L2 신규 테스트** `p2r3_stale_critical_lock_is_reclaimed_atomically`: `filetime`으로 락 mtime을 10초 백데이트 → 저장이 타임아웃 없이 즉시 성공 + `.claim.*` 잔여물 0 확인. (l2_tests dev-dependency에 `filetime` 추가)

### 14.6 R4 — changelog 사용자별 분리 (`main.rs`)
- `append_changelog`: `날짜.jsonl` → `날짜_{userId}.jsonl`(userId는 결함#8과 동일한 파일명 인코딩, `sync_core::encode_lock_id` 재사용 — pub 승격). **파일당 작성자 1명 = append 경합 원천 제거.**
- `read_changelog`: 해당 날짜의 레거시 `날짜.jsonl` + `날짜_*.jsonl` 전부 집계 후 timestamp 정렬 — 기존 로그 하위호환.

### 14.7 P3-1 — 저장 디바운스 설정값화 (`App.tsx`)
- 기본 1000ms 유지(동작 무변경). `app_config.saveDebounceMs`(하한 250ms)로 운영 중 조정 가능 — 10인 이상에서 저장 경합이 심하면 2000ms 등으로 완화(§9.2 T1).

### 14.8 검증 결과 (전 계층 회귀 없음)
| 계층 | 결과 | 비고 |
|---|---|---|
| L1 | ✅ 14/14 | O(n) 리팩터링 후 |
| L2 | ✅ 11/11 | P2-R3 신규 테스트 포함 |
| L3 | ✅ 10/10 | 아래 "검증 방법 변경" 참조 |
| L4 (UNC) | ✅ 14/14 | 〃 |
| 웹 (Playwright) | ✅ 31/31 | App.tsx 변경 회귀 없음 |

- **검증 방법 변경(중요):** 코얼레싱+폴링 폴백 도입 후, 동시 편집이 원시 VERSION_CONFLICT 없이 **선제 병합으로 해소**되는 경우가 정상 경로가 됐다(실행 로그에서 실제 관측 — "선제 병합으로 CAS 충돌 없이 해소"). 기존 L3/L4의 충돌 단언은 "VERSION_CONFLICT 문자열 관측"(메커니즘)이었는데, 이를 진짜 안전 속성인 **무손실 수렴 단언**(정착 후 A 화면=B 화면=파일, 두 마커 중 하나 생존, 무음 발산 없음)으로 전환했다. 충돌 해소 경로(CAS 경유 vs 선제 병합)는 정보성 로그로 유지.
- 부수 효과 실측: L3/L4 동시 편집 시나리오에서 VERSION_CONFLICT 발생 0건(수정 전에는 매회 1~3건) — 코얼레싱+폴링이 충돌 빈도 자체를 낮춤(§9.2 T2 완화).

### 14.9 권고안 (미실행 — 위험/대규모)
1. **R5 완전판(보류 변경 재시작 병합):** 마지막 동기화 payload를 localStorage에 영속하고 재시작 시 3-way 병합. init 흐름(8곳의 lastSavedPayload 대입) 전면 수정이 필요해 회귀 위험 대비 이득(수 초의 유실 창)이 작음 — 필요 시 별도 작업 권고.
2. **번들 축소:** `dist/index.html` ~2.0MB는 vite-singlefile에 의한 **의도적** 단일 파일(오프라인 HTML 내보내기/배포 요구). 코드 스플리팅은 이 요구와 상충 — 현상 유지 권고. 필요 시 lucide-react 아이콘 개별 임포트 정리로 소폭 절감 가능.
3. **파일 분할/델타 로그(장기):** 전체 파일 재작성 모델의 근본 한계는 §4 대안3(append-only 델타)로만 해소 가능 — 데이터 포맷·병합·복구 전면 재설계이므로 별도 프로젝트로 권고.
4. **웹 모드(server.ts) 동일 최적화:** 현재 프론트가 해당 API를 호출하지 않아(§7.3) 실익 없음 — 웹↔네이티브 공유파일 연동 시점에 반영 권고.
